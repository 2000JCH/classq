# ClassQ 서비스 개요

## 프로젝트 목적

대학교 수강신청 폭주 상황에서 발생하는 RDS 부하 문제를 해결하기 위한 시스템이다. 수강신청 요청이 몰리는 동기 구간에서 RDS를 완전히 제거하는 것이 핵심 목표이며, 이를 위해 Redis로 잔여 자리를 빠르게 확보하고 Kafka로 후속 DB 처리를 비동기화하는 구조를 채택한다.

---

## 사용자

시스템은 학생, 교수, 관리자 세 가지 역할을 가진 사용자로 구성된다. 학생은 수강신청, 취소, 대기자 등록을 수행하고, 교수는 강의 등록, 수정, 폐강을 담당하며, 관리자는 전체 시스템을 관리한다. 역할 구분은 JWT의 role 컬럼(STUDENT / PROFESSOR / ADMIN)으로 처리한다.

---

## 도메인 규칙

### 학생

학생의 학년은 정수 컬럼으로 관리한다. 유급이나 초과학기 학생을 고려하여 상한을 두지 않는다. 모든 학생은 반드시 학과에 소속되어야 한다.

### 강의

강의는 course_type(MAJOR_REQUIRED / MAJOR_ELECTIVE / LIBERAL_ARTS), class_type(THEORY / PRACTICE), class_mode(ONLINE / OFFLINE)로 구분된다. 학점은 교수가 강의 등록 시 1 / 2 / 3 중 선택한다. 전공 강의는 학과 제한(department_id 필수)과 학년 제한(min_grade / max_grade)이 있으며, 교양 강의는 학과 및 학년 제한이 없다(department_id = NULL, min_grade = NULL, max_grade = NULL). 강의 정보는 변경 빈도가 낮으므로 Redis 캐싱 없이 RDS에서 직접 조회한다.

### 수강신청

학생 1인당 최대 19학점까지 신청 가능하다. 수강신청 시 학점 초과 체크와 시간표 중복 체크를 Redis에서 수행하여 RDS 조회를 제거한다. 캐시가 없을 때만 RDS에서 시간표와 총 학점을 한 번에 조회한 후 Redis에 저장한다. 시간표 중복 기준은 일부라도 겹치면 거절이며, 딱 끝나는 시간은 겹침으로 보지 않는다(09:00~11:00 / 11:00~13:00 통과). 수강신청 취소는 가능하며, Redis 캐시는 동기 구간에서 즉시 갱신한다. 학년별 수강신청 날짜 분리는 추후 개선 사항이다.

### 대기자

수강 자리가 없을 때만 대기 등록 옵션을 노출한다. 대기 자리도 없으면 옵션 자체를 노출하지 않는다. 학생 1인당 최대 3개 과목 대기 가능하며, 과목당 최대 대기자 수는 교수가 강의 등록 시 waitlist_limit 컬럼으로 설정한다. 자리가 나면 대기 순번 1번부터 알림을 발송하고 잠금을 세팅한다. 대기자는 알림을 받은 후 10분 내에 수락해야 하며, 초과 시 Scheduler가 다음 순번으로 넘긴다. 수락 시 학점 초과 또는 시간 중복이면 자동으로 다음 순번으로 넘어간다.

### 잠금 로직

수강 자리가 0개이고 대기자 처리 중일 때만 잠금을 세팅한다. 잠금 중 신청 시도 시 "현재 대기자 처리 중입니다"로 응답하며, 대기 등록 옵션은 제공하지 않는다. 잠금 해제 조건은 대기자 수락 완료, 모든 대기자 만료 후 다음 순번 없음, 강의 폐강 세 가지다.

## Redis 캐싱 전략

Redis는 수강신청 동기 구간에서 RDS 부하를 제거하기 위해 사용한다. 잔여 수강 자리(enrollment:course:{id})는 강의 등록 시 정원으로 초기화하며, key가 없을 때 RDS에서 자동 복구한다. 잔여 대기 자리(waitlist:course:{id})는 강의 등록 시 waitlist_limit으로 초기화한다. 학생 시간표(schedule:student:{id})와 총 학점(credits:student:{id})은 첫 수강신청 시 RDS에서 조회 후 저장하며 이후 수강신청/취소 시 즉시 갱신한다. 잠금(lock:course:{id})은 TTL 없이 세팅하며 Scheduler가 직접 관리한다. Redis AOF 설정을 활성화하여 장애 시 디스크에서 데이터를 복원할 수 있도록 한다.

## 기술 스택

### 인프라

EKS를 사용하여 Spring Boot, Kafka, Debezium 컨테이너를 배포 및 관리한다. ECR에 Docker 이미지를 저장하고 버전을 관리한다. RDS(MySQL)를 영구 데이터 저장소로 사용한다.

### 캐시

- 로컬 개발: Redis
- AWS 배포: ElastiCache (Redis)
- 로컬과 AWS 모두 동일한 Redis 엔진을 사용하며 애플리케이션 코드 변경 없이 전환 가능하다.

### 데이터 파이프라인

Kafka를 수강신청/취소 이벤트의 비동기 처리 브로커로 사용한다. Debezium이 course 테이블 변경을 감지하여 Kafka로 발행하는 CDC 역할을 담당한다.

### 보안

JWT 인증을 사용하며 role 컬럼(STUDENT / PROFESSOR / ADMIN)으로 권한을 구분한다.

### 모니터링

Prometheus가 애플리케이션 메트릭을 수집하고, CloudWatch가 AWS 인프라 메트릭을 수집한다. Grafana는 두 소스를 통합한 대시보드를 제공한다.

## 테이블 목록

시스템은 account, student, professor, department, course, course_schedule, enrollment, waitlist, notification 총 9개 테이블로 구성된다.

## 추후 개선 사항

학년별 수강신청 날짜 분리(1~2학년 / 3~4학년), 대기자 학년별 슬롯 분리, 배치 잡으로 수강신청 시작 전 Redis 캐시 워밍업, 부하 테스트 후 병목 구간 개선, Flink 도입을 통한 수강신청 실시간 집계/분석이 예정되어 있다.
