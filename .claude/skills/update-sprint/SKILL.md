# update-sprint

Sprint work-log 파일(`.claude/work-log/sprint{N}-phase{N}.md`)을 정해진 템플릿 형식으로 작성하거나 업데이트한다.

---

## 실행 절차

### 1단계 — 대상 확인

사용자 메시지에서 Sprint 번호를 먼저 파악한다. 번호가 명시되어 있으면 바로 대상 파일을 결정하고, 없을 때만 아래 질문을 한다.

> 몇 번째 Sprint를 업데이트할까요? (예: 1, 2, 3, 4)

대상 파일 규칙:
- Sprint 1 → `.claude/work-log/sprint1-phase1.md`
- Sprint 2 → `.claude/work-log/sprint2-phase2.md`
- Sprint 3 → `.claude/work-log/sprint3-phase3.md`
- Sprint 4 → `.claude/work-log/sprint4-phase4.md`
- 그 이상도 같은 규칙(`sprint{N}-phase{N}.md`)으로 처리한다.

### 2단계 — 현재 파일 읽기

대상 `.md` 파일을 Read 도구로 읽어 현재 내용을 파악한다.

### 3단계 — 커밋 내역 확인

파일에 기록된 **기간(기간: YYYY-MM-DD ~ YYYY-MM-DD)** 을 기준으로 git log를 조회한다.

```
git log --oneline --after="YYYY-MM-DD" --before="YYYY-MM-DD" --reverse
```

기간이 비어 있거나 예정 상태면 최근 커밋 20개를 조회해 관련 커밋을 추려낸다.

```
git log --oneline -20
```

커밋 내역을 바탕으로 실제 구현 내용, 트러블슈팅 항목을 파악한다.

### 4단계 — 파일 작성

아래 템플릿을 기준으로 파일을 업데이트한다. 각 항목 작성 규칙은 **템플릿 규칙** 섹션을 따른다.

파일이 이미 존재하는 경우:
- 기존 내용을 먼저 읽고 커밋 내역과 비교한다.
- 이미 작성된 내용은 중복으로 추가하지 않는다.
- 누락된 내용과 잘못된 내용만 추가하거나 수정한다.
- 애매한 부분이 있으면 작성 전에 사용자에게 질문한다.


---

## 템플릿

```markdown
# Sprint {N} — Phase {N}. {Phase 제목}

**기간:** YYYY-MM-DD ~ YYYY-MM-DD  
**상태:** ✅ 완료 / 🔄 진행 중 / ⬜ 예정

---

## 목표

**배경**  
이 Phase가 왜 필요한지 — 전체 시스템에서 어떤 역할을 하는지 서술한다.

**문제 정의**  
이전 Phase에서 무엇이 없었고, 그것이 왜 문제인지 서술한다.

**대상 사용자/화면**  
인프라/설정 단계면 "해당 없음 (인프라 설정 단계)"으로 작성한다.

**성공 기준 (완료 판정)**
- 기능: 어떤 흐름이 정상 동작해야 하는가
- 성능: 특정 수치 기준이 있는 경우 기재, 없으면 "해당 없음"
- 정확도: 데이터 정합성, 보안 조건 등
- 메모리: Redis 등 인메모리 상태 조건, 없으면 "해당 없음"

## 구현

각 컴포넌트(서비스, 필터, 유틸 등)별로 어떻게 만들었는지 서술한다.
커밋 내역을 참고해 실제 구현 사항을 채운다.
```

## 템플릿 규칙

| 항목 | 규칙 |
|---|---|
| 처리 대상 | 해당 없으면 섹션 자체를 생략 (해당 없음 문구 사용 금지) |
| 트러블슈팅 | 문제가 없으면 섹션 자체를 생략 |
| 대상 사용자/화면 | 인프라·설정 단계는 "해당 없음"으로 작성 |
| 성공 기준 | 성능/메모리가 없으면 "해당 없음"으로 작성 |
| 제공 인터페이스 | 권한 컬럼 — 인증 불필요면 "없음", 인증 필요면 "인증 필요", 역할 필요면 "STUDENT/PROFESSOR/ADMIN" |
| 상태 | 완료 ✅ / 진행 중 🔄 / 예정 ⬜ |
| 구현 섹션 | git log 기반으로 실제 구현 사항을 채운다. 예정 상태면 "작업 시작 후 채워넣기"로 비워둔다. |
